# Capstone Architecture Design

## Storage Systems

### Goal 1 — Predict Patient Readmission Risk
**Storage: Data Warehouse (Amazon Redshift or Google BigQuery)**

Historical treatment data — diagnoses, procedures, medications, discharge summaries, and lab results — is structured and relational. A data warehouse is the correct home for this data because it supports complex multi-table analytical queries needed to extract features for the ML model (e.g., "number of ER visits in last 6 months," "average length of stay by diagnosis code"). The warehouse also serves as the feature store from which the readmission prediction model is trained and re-trained periodically.

### Goal 2 — Allow Doctors to Query Patient History in Plain English
**Storage: Vector Database (Pinecone or Weaviate) + Document Store (MongoDB)**

Patient history notes, discharge summaries, and clinical narratives are unstructured text. These documents are chunked, embedded using a clinical language model (such as BioBERT or ClinicalBERT), and stored as vectors in a vector database. When a doctor asks "Has this patient had a cardiac event before?", the query is embedded and matched semantically against stored document chunks — retrieving relevant records regardless of exact phrasing. MongoDB serves as the underlying document store for full document retrieval once the relevant chunks are identified.

### Goal 3 — Generate Monthly Management Reports
**Storage: Data Warehouse (same instance as Goal 1)**

Bed occupancy rates, department-wise costs, and resource utilization are structured metrics aggregated from the OLTP systems. These reports are generated by scheduled SQL queries on the data warehouse, which already holds cleaned historical data. BI tools like Apache Superset or Tableau connect directly to the warehouse to render dashboards for hospital management.

### Goal 4 — Stream and Store Real-Time ICU Vitals
**Storage: Time-Series Database (InfluxDB or Apache Cassandra)**

ICU vitals (heart rate, blood pressure, SpO2, temperature) are generated at high frequency — potentially every second per patient. A time-series database is purpose-built for this: it handles high-throughput sequential writes, supports downsampling for long-term storage, and enables efficient range queries like "show me this patient's heart rate for the last 4 hours." Apache Kafka acts as the streaming ingestion layer, feeding data into InfluxDB in real time.

## OLTP vs OLAP Boundary

The **OLTP boundary** encompasses all systems that handle real-time clinical operations: the Hospital Information System (HIS) recording admissions, discharges, and prescriptions; the ICU monitoring system writing vitals to InfluxDB; and the Electronic Health Record (EHR) storing patient notes in MongoDB. These systems prioritize write performance, low latency, and ACID compliance for transactional integrity.

The **OLAP boundary** begins at the ETL pipeline. Nightly batch jobs extract cleaned, transformed data from the OLTP systems and load it into the Data Warehouse. The warehouse is the single source of truth for all analytical workloads — readmission model training, management reporting, and department cost analysis. No analytical query touches the OLTP systems directly, protecting operational performance. The vector database sits at the OLAP boundary as well, serving semantic search over historical documents that have already been processed and indexed offline.

---

## Trade-offs

**Trade-off: Data Freshness vs. Analytical Isolation**

The most significant trade-off in this design is the nightly batch ETL between OLTP and OLAP systems. Because the data warehouse is refreshed once per night, management reports and the readmission model's training data lag behind real-world events by up to 24 hours. A patient discharged at 11 PM would not appear in the next morning's occupancy report.

**Mitigation:** For the readmission prediction model specifically, batch latency is acceptable — the model predicts risk based on historical patterns, not live vitals. For management reporting, near-real-time reporting requirements (if they emerge) can be addressed by implementing Change Data Capture (CDC) using Apache Debezium, which captures row-level changes from the EHR database and streams them into the warehouse incrementally rather than in a nightly dump. This reduces lag from 24 hours to under 15 minutes without abandoning the OLTP/OLAP separation that protects clinical system performance. The ICU vitals goal is already served by a real-time path (Kafka + InfluxDB), so it is unaffected by this trade-off.
